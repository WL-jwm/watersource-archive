## 一、现状分析

### 计算引擎现状（zoneCalcEngine.ts，546 行）

| 模块 | 现状 | 缺口 |
|------|------|------|
| 地下水经验值法 | 完整，支持孔隙水/裂隙水/岩溶水三级保护区 | 无 |
| 地下水解析法 | 完整，Cooper-Jacob 公式 + 流速法取大值 | 无 |
| 地表水河流型 | 基础框架，仅按流量/河宽判断规模取经验值 | 缺少潮汐河段、分支流汇入口处理；二级保护区上溯距离计算粗糙；缺少纵向衰减系数 |

## 二、A1 地表水保护区划分计算

### 2.1 现状缺口

**河流型**：
- 规模判断仅依据流量和河宽，缺少河段特征（潮汐影响、分支汇入）
- 一级保护区下游距离固定取上游的 20%，HJ 338-2018 实际规定"不小于 100m"（小型河流）至"不小于 300m"（大型河流）
- 二级保护区上溯距离按一级的 3 倍估算，规范实际规定"从一级保护区上边界向上游延伸至不小于 X 米"
- 缺少两岸陆域纵深的坡度修正

**湖库型**：
- 规模分级仅按水面面积，缺少库容指标（HJ 338-2018 以库容为主要分级依据）
- 二级保护区面积估算过于简化，小型湖库应取"整个汇水区域"而非简单面积外扩
- 缺少取水口位置（岸边/湖心）对保护区形态的影响
- 缺少湖库型二维经验模型（扇形保护区）

**拐点坐标**：
- 河流型仅生成矩形，实际应为"沿河道走向的不规则带状"
- 湖库型复用圆形生成器，实际可能为扇形

### 2.3 A1-T1 河流型保护区计算增强

#### 2.3.1 规模分级增强

现状：仅按流量/河宽判断大/中/小型。

增强为多指标综合判断：

```typescript
interface RiverScaleParams {
  /** 多年平均流量 m³/s */
  meanFlow: number;
  /** 河道平均宽度 m */
  riverWidth: number;
  /** 河道平均水深 m */
  meanDepth?: number;
  /** 河床纵比降 ‰ */
  slope?: number;
  /** 是否受潮汐影响 */
  isTidal?: boolean;
  /** 是否有支流汇入（取水口上游 1000m 内） */
  hasTributary?: boolean;
}

type RiverScale = '大型' | '中型' | '小型';

function classifyRiverScale(params: RiverScaleParams): { scale: RiverScale; basis: string }
```

分级标准（依据 HJ 338-2018 第 5.2 节 + 河北省河流特征）：

| 指标 | 大型 | 中型 | 小型 |
|------|------|------|------|
| 多年平均流量 | ≥ 150 m³/s | 15~150 m³/s | < 15 m³/s |
| 河道宽度 | ≥ 200 m | 50~200 m | < 50 m |
| 河道水深 | ≥ 5 m | 2~5 m | < 2 m |

潮汐河段特殊处理：受潮汐影响的河段，一级保护区下游距离应取潮汐上溯距离与经验值的较大值。

#### 2.3.2 保护区距离计算修正

**一级保护区**（HJ 338-2018 第 5.2.1 节）：

| 规模 | 上游长度 | 下游长度 | 两岸纵深 |
|------|---------|---------|---------|
| 大型 | 不小于 5000 m | 不小于 300 m | 不小于 50 m（遇坡度>15° 增至 100 m） |
| 中型 | 不小于 3000 m | 不小于 200 m | 不小于 50 m |
| 小型 | 不小于 1000 m | 不小于 100 m | 不小于 50 m |

修正点：
1. 下游距离从"上游 × 20%"改为查表取规范值
2. 两岸纵深考虑坡度修正：`adjustedWidth = baseWidth + max(0, (slope - 15)) * 2`
3. 潮汐河段下游距离取 `max(规范值, 潮汐上溯距离)`

**二级保护区**（HJ 338-2018 第 5.2.2 节）：

| 规模 | 上游延伸（从一级边界起） | 下游延伸 | 两岸纵深 |
|------|----------------------|---------|---------|
| 大型 | 不小于 10000 m | 不小于 500 m | 不小于 1000 m |
| 中型 | 不小于 5000 m | 不小于 300 m | 不小于 500 m |
| 小型 | 不小于 2000 m | 不小于 200 m | 不小于 200 m |

修正点：
1. 上游距离从"一级 × 3"改为"从一级边界起独立计算延伸距离"
2. 两岸纵深独立取值，不再简单翻倍
3. 面积计算改为梯形/带状：`area = (上游总长度 + 下游总长度) × 两岸总宽度`

**准保护区**（HJ 338-2018 第 5.2.3 节）：

河流型准保护区为二级保护区上游侧延伸至汇水区边界。当无法确定汇水区时：
- 上游延伸距离 = 二级上游延伸 × 2
- 两岸纵深 = 二级两岸纵深 × 1.5
- 需明确标注"实际划定应以流域汇水分析为准"

#### 2.3.3 ZoneResult 输出增强

```typescript
// 河流型结果新增字段
interface RiverZoneExtension {

  /** 下游总长度（从取水口起算） */
```

### 2.4 A1-T2 湖库型保护区计算增强

#### 2.4.1 规模分级增强

```typescript
interface LakeScaleParams {
  /** 水面面积 km² */
  surfaceArea: number;
  /** 总库容 亿 m³ */
  totalCapacity?: number;
  /** 最大水深 m */
  maxDepth?: number;
  /** 平均水深 m */
  meanDepth?: number;
  /** 取水口类型 */
  intakeType?: '岸边' | '湖心' | '分层取水';
}

function classifyLakeScale(params: LakeScaleParams): { scale: '大型' | '中型' | '小型'; basis: string }
```

分级标准（HJ 338-2018 第 5.3 节）：

| 指标 | 大型 | 中型 | 小型 |
|------|------|------|------|
| 总库容 | ≥ 10 亿 m³ | 1~10 亿 m³ | < 1 亿 m³ |
| 水面面积 | ≥ 50 km² | 5~50 km² | < 5 km² |

当库容和面积不一致时，取较高级别（更保守）。

#### 2.4.2 保护区计算增强

**一级保护区**（HJ 338-2018 第 5.3.1 节）：

| 规模 | 取水口半径 | 附加要求 |
|------|----------|---------|
| 小型 | 取水口半径 300 m 范围内的水域 | 陆域为水域外延 50 m |
| 中型 | 取水口半径 500 m 范围内的水域 | 陆域为水域外延 50 m |
| 大型 | 取水口半径 1000 m 范围内的水域 | 陆域为水域外延 50 m |

修正点：现有代码取 1000/2000/3000 m 偏大，修正为规范值 300/500/1000 m。增加陆域外延 50 m 的描述。

**二级保护区**（HJ 338-2018 第 5.3.2 节）：

| 规模 | 水域范围 | 陆域范围 |
|------|---------|---------|
| 小型 | 整个水域面积 | 水域外延不小于 1000 m |
| 中型 | 一级保护区外不小于 1000 m 范围的水域 | 水域外延不小于 1000 m |
| 大型 | 一级保护区外不小于 2000 m 范围的水域 | 水域外延不小于 2000 m |

修正点：现有代码二级面积估算过于简单，改为根据规模分别计算水域和陆域面积。

**准保护区**（HJ 338-2018 第 5.3.3 节）：

湖库型准保护区为二级保护区外至分水岭/汇水区边界。当无法确定时：
- 按二级保护区面积 × 1.5 估算
- 需标注"实际划定应以流域地形分析为准"

#### 2.4.3 取水口位置影响

| 取水口类型 | 保护区形态 | 说明 |
|-----------|----------|------|
| 岸边 | 半圆（向水域侧） | 陆域侧仅含沿岸带 |
| 湖心 | 全圆 | 水域为完整圆形 |
| 分层取水 | 全圆 + 深度说明 | 需注明取水层深度 |

在 `CalcParams` 中增加 `intakeType?: '岸边' | '湖心' | '分层取水'`，影响一级保护区面积计算（岸边取水面积减半）。

### 2.5 A1-T3 拐点坐标生成适配

#### 2.5.1 河流型拐点生成

现状：`generateRectVertices()` 生成规则矩形。

增强：生成"沿河道走向的带状保护区"拐点。

```typescript
/**
 * 生成河流型保护区拐点
 * 
 * 当无河道走向数据时，简化为以取水口为中心的矩形（上游长 + 下游长，两岸各宽）
 * 当有河道走向数据时，沿河道中心线两侧缓冲生成拐点
 *
 * @param centerLng 取水口经度
 * @param centerLat 取水口纬度
 * @param upstreamLength 上游长度 m
 * @param downstreamLength 下游长度 m
 * @param bankWidth 两岸单侧宽度 m
 * @param riverDirection 河道流向方位角（度，从正北顺时针），默认 0（正北→正南流向）
 */
export function generateRiverVertices(
  centerLng: number,
  centerLat: number,
  upstreamLength: number,
  downstreamLength: number,
  bankWidth: number,
  riverDirection?: number,
): ZoneVertex[]
```

输出 8 个拐点（矩形四角 + 上下游中点 + 取水口两侧），标注各点方位描述。

#### 2.5.2 湖库型拐点生成

现状：复用 `generateCircleVertices()`。

增强：根据取水口类型生成不同形态：

```typescript
/**
 * 生成湖库型保护区拐点
 *
 * @param intakeType 取水口类型
 * - 岸边：半圆（水域侧），陆域为沿岸带
 * - 湖心：全圆
 * - 分层取水：全圆
 */
export function generateLakeVertices(
  centerLng: number,
  centerLat: number,
  radiusM: number,
  intakeType: '岸边' | '湖心' | '分层取水',
  vertexCount?: number,
): ZoneVertex[]
```

岸边取水：生成半圆（180°~360°范围，即水域侧），陆域侧标注沿岸带 50 m 外延。

#### 2.5.3 generateBatchVertices 适配

`generateBatchVertices()` 内部根据 `ZoneResult` 的 `method` 和 `level` 字段判断调用哪个生成函数。增加对河流型和湖库型的分支判断。

## 三、B1 报告生成体系升级

### 3.1 现状缺口

1. **章节固定**：四章结构不可选择，用户无法按需生成
2. **单一模板**：无法在简版（仅结果表）/标准版（含计算过程）/详版（含图件+敏感性分析）间切换
3. **无图件**：报告纯文字+表格，缺少地图截图和保护区可视化图
4. **仅 Word**：无 PDF 输出，部分场景需要 PDF 直接归档
5. **无编号**：报告无唯一编号，难以管理和引用

### 3.2 任务拆分

| 任务编号 | 名称 | 涉及文件 | 预估行数 |
|---------|------|---------|---------|
| B1-T1 | 报告章节勾选与多模板 | zoneReportGenerator.ts | +300 行（重构） |
| B1-T2 | 图件嵌入 | zoneReportGenerator.ts + ProtectionZoneCalc.tsx | +200 行 |

### 3.4 B1-T2 图件嵌入

#### 3.4.1 图件类型

| 图件 | 来源 | 用途 |
|------|------|------|
| 地图截图 | MapView 页面 html2canvas | 展示水源地分布和保护区范围 |
| 保护区可视化图 | ProtectionZoneCalc 结果卡片 | 展示单个水源地保护区边界 |
| 敏感性分析图 | ProtectionZoneCalc 敏感性面板 | 展示参数-面积影响曲线（详版） |

#### 3.4.2 实现方案

**方案**：在生成报告前，调用 html2canvas 对指定 DOM 节点截图，转为 ImageRun 嵌入 Word。

```typescript
import { ImageRun } from 'docx';

/**
 * 将 DOM 元素截图并转为 ImageRun
 * @param element 目标 DOM 元素
 * @param width 图片宽度（EMU，默认 600 万 = 约 15cm）
 */
async function domToImageRun(element: HTMLElement, width: number = 6000000): Promise<ImageRun>
```

**流程**：
1. 用户在 MapView 页面调整好视图（水源地分布 + 保护区边界）
2. 点击"生成报告"时，先对地图 DOM 截图
3. 对当前结果卡片中的保护区可视化区域截图
4. 将截图数据传入 `generateZoneReport()`，在对应章节插入

**限制说明**：
- 地图瓦片需确保 `crossOrigin` 属性已设置（MapView 已有 P4-14 的瓦片跨域截图方案）
- 截图操作需要用户在页面上预览好视图后触发，不能自动定位
- 在报告配置弹窗中增加提示："请先在地图页面调整好视图，截图将使用当前地图显示范围"

#### 3.4.3 Word 中图件布局

```typescript
// 图片居中，带图注
function imageWithCaption(imageRun: ImageRun, caption: string): (Paragraph | Table)[] {
  return [
    new Paragraph({
      children: [imageRun],
      alignment: AlignmentType.CENTER,
      spacing: { before: 200, after: 80 },
    }),
    new Paragraph({
      children: [new TextRun({ text: caption, size: 18, font: 'SimSun', color: '666666', italics: true })],
      alignment: AlignmentType.CENTER,
      spacing: { after: 200 },
    }),
  ];
}
```

图件插入位置：
- 第一章概述末尾：水源地分布总图
- 第三章逐个水源地详情中：该水源地保护区可视化图
- 详版敏感性分析章节：敏感性曲线图

### 3.5 B1-T3 PDF 导出

#### 3.5.1 技术选型

| 方案 | 优点 | 缺点 | 选择 |
|------|------|------|------|
| docx 生成 Word → 浏览器打印为 PDF | 保留 Word 格式 | 需用户手动操作 | ❌ |
| html2canvas + jsPDF | 纯前端，可控 | 分页困难，图片质量受屏幕分辨率限制 | 备选 |
| docx 生成 Word + Browser print API | 格式精确 | 需用户交互选择"另存为 PDF" | ✅ 主方案 |

**主方案**：生成 .docx 后自动打开打印预览（`window.open()` + Blob URL），用户选择"另存为 PDF"。

**备选方案**：当仅需快速导出（不需要精确排版）时，使用 jsPDF 直接生成 PDF：

```typescript
// 新建 src/lib/reportPdfExporter.ts
import { jsPDF } from 'jspdf';

/**
 * 快速 PDF 导出（简化排版）
 * 适用于仅需结果表格、不需要复杂排版的场景
 */
export async function generatePdfReport(
  results: ZoneCalcRecord[],
  sources: WaterSourceRecord[],
  config: ReportConfig,
): Promise<void>
```

PDF 排版规则：
- A4 纵向，页边距 15mm
- 标题：18pt 黑体居中
- 正文：10pt 宋体
- 表格：8pt，带边框
- 分页：每章新起一页

#### 3.5.2 UI 交互

报告配置弹窗中"导出格式"选择：
- ☑ Word(.docx)：调用 `generateZoneReport()`
- ☐ PDF(.pdf)：调用 `generatePdfReport()`（备选方案，快速导出）
- ☐ Word + PDF：同时导出两种格式

选择 Word + PDF 时先生成 Word，延迟 500ms 后生成 PDF。
