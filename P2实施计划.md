## 二、P2-1：静态数据改为 IDB 优先加载

### 2.1 目标

将 653KB 的 `data-sources` chunk 从首屏阻塞加载改为按需/懒加载，预计首屏 JS 体积减少约 500KB+（gzip 约 65KB），首屏渲染时间缩短 30~50%。

### 2.2 核心思路

```
当前流程：  静态 import → 打包入 data-sources.js → 首屏阻塞加载
                              ↓
目标流程：  IDB 有数据 → 直接从 IDB 读取（零网络开销）
          IDB 无数据 → 动态 import() 静态数据 → 写入 IDB → 再读取
```

### 2.3 实施步骤

#### 步骤 1：改造 waterSourceStore.ts 的数据加载逻辑

**文件**：`src/stores/waterSourceStore.ts`

将顶部的静态 import 改为动态 import：

```typescript
// 当前（静态 import，打包入首屏 chunk）
import { hebeiWaterSources } from '@/data/hebeiWaterSources';
import { waterSourceGeo } from '@/data/waterSourceGeoData';

// 改为：移除顶部静态 import，在 initDB 内部按需动态 import
async function loadStaticData() {
  const { hebeiWaterSources } = await import('@/data/hebeiWaterSources');
  const { waterSourceGeo } = await import('@/data/waterSourceGeoData');
  return { hebeiWaterSources, waterSourceGeo };
}
```

修改 `initDB` 函数中 `count === 0 || count < 500` 分支，调用 `loadStaticData()` 替代直接引用。

修改 `buildRecordsFromStatic` 函数，改为接收参数而非闭包引用：

```typescript
// 当前：直接引用模块级变量
function buildRecordsFromStatic() { ... hebeiWaterSources ... }

// 改为：接收参数
function buildRecordsFromStatic(data: HebeiWaterSource[]) { ... }
```

#### 步骤 2：改造 Home.tsx 的数据引用

**文件**：`src/pages/Home.tsx`

Home.tsx 直接 import 了 `hebeiWaterSources`、`hebeiDivisions`、`sampleData`。改为从 store 获取已有数据，或对一次性统计函数做懒加载：

```typescript
// 当前
import { hebeiWaterSources, getHebeiWaterSourceStats, getCityTownshipSources } from '@/data/hebeiWaterSources';

// 改为：从 store 获取（store 已在 Layout.tsx 中初始化 IDB）
const { sources } = useWaterSourceStore();
// 统计函数改为基于 store 数据计算，或懒加载
```

对于 `sampleData`（报告示例数据），改为路由级懒加载（仅在访问首页报告详情时加载）。

#### 步骤 3：改造 DivisionSelector / DivisionOverview 的数据引用

**文件**：`src/components/DivisionSelector.tsx`、`src/pages/DivisionOverview.tsx`

`hebeiDivisions.ts`（26KB）体积较小，可保留静态 import。`hebeiTownships.ts`（370KB）改为动态 import：

```typescript
// DivisionOverview.tsx 首屏不需要乡镇数据时
const [townshipData, setTownshipData] = useState(null);
useEffect(() => {
  import('@/data/hebeiTownships').then(m => setTownshipData(m.default));
```

#### 步骤 4：清理未使用的 JSON 文件

**文件**：`src/data/hebei-divisions.json`、`src/data/hebei-townships.json`、`src/data/waterSourceGeo.json`

这三个 JSON 文件（共 554KB）未被任何 TS 文件 import，是早期 TS 化前的遗留数据。确认无外部引用后移至 `public/data/` 目录或删除。

**文件**：`vite.config.ts`

## 三、P2-2：暗色模式迁移到 CSS 变量

### 3.1 目标

将 130 行 `.dark .xxx { ... !important }` 覆盖规则替换为 CSS 变量切换方案，消除 `!important` 依赖，使暗色模式维护从"逐元素覆盖"变为"变量赋值"。

### 3.2 核心思路

```
当前方案：  :root { --color-surface: #FFFFFF; }
            .dark .bg-white { background-color: #1E293B !important; }  ← 覆盖

目标方案：  :root { --color-surface: #FFFFFF; --color-surface-2: #F8FAFC; ... }
            .dark { --color-surface: #1E293B; --color-surface-2: #0F172A; ... }  ← 仅切换变量
            .bg-surface { background-color: var(--color-surface); }   ← 自动适配
```

### 3.3 实施步骤

#### 步骤 1：扩展 `:root` CSS 变量，新增 `.dark` 变量覆盖

**文件**：`src/index.css`

在现有 `:root` 块后添加 `.dark` 选择器，重新声明暗色变量值：

```css
:root {
  --color-surface: #FFFFFF;
  --color-surface-secondary: #F8FAFC;
  --color-surface-tertiary: #F1F5F9;
  --color-border: #E2E8F0;
  --color-text-primary: #1E293B;
  --color-text-secondary: #64748B;
  --color-text-tertiary: #94A3B8;
  /* ... 其他现有变量保持不变 ... */
}

.dark {
  --color-surface: #1E293B;
  --color-surface-secondary: #0F172A;
  --color-surface-tertiary: #334155;
  --color-border: #475569;
  --color-text-primary: #E2E8F0;
  --color-text-secondary: #94A3B8;
  --color-text-tertiary: #64748B;
}
```

#### 步骤 2：新增语义化 CSS 工具类

**文件**：`src/index.css`

为常用颜色添加语义化类名，直接引用 CSS 变量：

```css
@layer components {
  .bg-surface { background-color: var(--color-surface); }
  .bg-surface-secondary { background-color: var(--color-surface-secondary); }
  .bg-surface-tertiary { background-color: var(--color-surface-tertiary); }
  .text-text-primary { color: var(--color-text-primary); }
  .text-text-secondary { color: var(--color-text-secondary); }
  .text-text-tertiary { color: var(--color-text-tertiary); }
  .border-surface-border { border-color: var(--color-border); }
}
```

> 注意：部分语义化类名已在 `tailwind.config.js` 的 `colors.extend` 中定义，需确保两者不冲突。Tailwind 的 `surface`、`text` 色板已映射到固定 hex 值，不随 `.dark` 切换。方案是：将 Tailwind 的 surface/text 色板改为引用 CSS 变量。

#### 步骤 3：修改 tailwind.config.js 色板引用 CSS 变量

**文件**：`tailwind.config.js`

```javascript
// 当前（固定值，暗色模式不切换）
surface: {
  DEFAULT: '#FFFFFF',
  secondary: '#F8FAFC',
  tertiary: '#F1F5F9',
  border: '#E2E8F0',
},

// 改为（引用 CSS 变量，暗色模式自动切换）
surface: {
  DEFAULT: 'var(--color-surface)',
  secondary: 'var(--color-surface-secondary)',
  tertiary: 'var(--color-surface-tertiary)',
  border: 'var(--color-border)',
},
text: {
  primary: 'var(--color-text-primary)',
  secondary: 'var(--color-text-secondary)',
  tertiary: 'var(--color-text-tertiary)',
},
```

这样所有使用 `bg-surface`、`text-text-primary` 等类名的组件**无需任何修改**即可自动适配暗色模式。

#### 步骤 4：删除 index.css 中的 `.dark .xxx { !important }` 覆盖规则

**文件**：`src/index.css`（第 207~330 行）

删除以下覆盖规则块（约 130 行）：

- `.dark .bg-white`、`.dark .bg-gray-50`、`.dark .bg-gray-100` 等背景色覆盖
- `.dark .text-gray-800`、`.dark .text-gray-700` 等文字色覆盖
- `.dark .border-gray-200`、`.dark .border-gray-300` 等边框色覆盖
- `.dark input[type="text"]` 等表单元素覆盖

保留以下规则（非颜色变量可覆盖的）：

- `.dark .leaflet-tile { filter: ... }`（地图瓦片暗色滤镜，无法用变量实现）
- `.dark .leaflet-popup-content-wrapper`（Leaflet 内部样式覆盖）
- `.dark body { background: ... }`（body 全局背景，可改为变量但需额外处理）

#### 步骤 5：处理页面组件中的硬编码颜色

**文件**：各 `src/pages/*.tsx`

搜索并替换组件中直接使用的 `bg-white`、`text-gray-800` 等 Tailwind 原生色名，改为语义化类名：

| 当前类名 | 替换为 | 说明 |
|---------|--------|------|
| `bg-white` | `bg-surface` | 卡片/面板背景 |
| `bg-gray-50` | `bg-surface-secondary` | 次要背景 |
| `bg-gray-100` | `bg-surface-tertiary` | 第三级背景 |
| `text-gray-800` | `text-text-primary` | 主要文字 |
| `text-gray-600` | `text-text-secondary` | 次要文字 |
| `text-gray-500` | `text-text-tertiary` | 辅助文字 |
| `text-gray-400` | `text-text-tertiary` | 辅助文字 |
| `border-gray-200` | `border-surface-border` | 边框 |

> 这一步工作量较大但机械性强，可编写脚本批量替换。需注意不替换 `bg-blue-600`、`text-red-500` 等功能性颜色（这些在暗色模式下保持不变或单独处理）。

#### 步骤 6：更新 badge 暗色规则

**文件**：`src/index.css`

将 `.dark .badge-success` 等规则改为 CSS 变量：

```css
:root {
  --badge-success-bg: rgba(16,185,129,0.1);
  --badge-success-text: #059669;
  /* ... */
}
.dark {
  --badge-success-bg: rgba(6,78,59,0.4);
  --badge-success-text: #34D399;
  /* ... */
}
.badge-success { background-color: var(--badge-success-bg); color: var(--badge-success-text); }
```

### 3.4 验收标准

| 指标 | 当前值 | 目标值 |
|------|--------|--------|
| index.css 行数 | 444 行 | < 320 行 |
| `!important` 数量 | ~80 处 | < 5 处（仅 Leaflet 特殊覆盖） |
| `.dark .xxx` 覆盖规则 | ~60 条 | < 10 条（仅 Leaflet + body） |
| 暗色模式视觉一致性 | 部分元素未适配 | 全部页面暗色正常 |
| 全部测试通过 | 157/157 | 157/157 |

### 3.5 风险与对策

| 风险 | 影响 | 对策 |
|------|------|------|
| Tailwind 色板改为 CSS 变量后，`opacity` 修饰符（如 `bg-surface/50`）可能失效 | 半透明效果丢失 | 对需要透明度的场景使用 `color-mix()` 或保留固定色值 |
| 批量替换类名可能遗漏个别组件 | 个别元素暗色模式异常 | 逐页面目视检查 + 截图对比 |
| Leaflet 地图组件内部样式不受 CSS 变量控制 | 地图暗色模式保持现有覆盖 | 保留 `.dark .leaflet-*` 规则不动 |

**1.5 天**（6 步骤，步骤 5 批量替换约 0.5 天，其余各 1~2 小时）

**文件**：`.github/workflows/ci.yml`

```yaml

  push:
    branches: [main, develop]
  pull_request:
    branches: [main]

  quality:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Setup Node.js
        uses: actions/setup-node@v4
        with:
          node-version: '20'
          cache: 'npm'

      - name: Install dependencies
        run: npm ci

      - name: TypeScript type check

      - name: ESLint
        run: npm run lint

      - name: Prettier check
        run: npm run format:check

      - name: Unit tests
        run: npm run test

      - name: Build
        run: npm run build

      - name: Upload build artifacts
        uses: actions/upload-artifact@v4
        with:
          name: dist
          path: dist/
          retention-days: 7
```
